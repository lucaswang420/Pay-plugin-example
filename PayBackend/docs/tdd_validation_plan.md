# TDD 验证计划 - Pay Plugin 重构

## 背景

Pay Plugin 重构已完成，但没有严格遵循 TDD 原则（先写测试）。
本计划用于验证现有代码和测试的价值。

## TDD 原则检查清单

### ❌ 违反的原则

- [ ] 每个函数/方法都有测试？
- [ ] 在实现前看测试失败？
- [ ] 测试因预期原因失败（功能缺失，而非拼写错误）？
- [ ] 写最小代码通过每个测试？
- [ ] 所有测试都通过？
- [ ] 输出干净（无错误、警告）？

**当前状态：** 我们违反了 #2 和 #3（没有看测试失败）

## 验证策略

### 阶段 1：测试发现（1小时）

**目标：** 确定哪些测试能真正验证行为

```
1. 尝试编译现有单元测试
   - test/service/PaymentServiceTest.cc
   - test/service/RefundServiceTest.cc
   - test/service/CallbackServiceTest.cc

2. 分析结果：
   a) 编译失败
      → 检查是否是测试问题还是代码问题
      → 如果测试代码错误，修复并重新编译

   b) 编译成功，测试失败
      → 检查失败原因
      → 如果测试正确发现了 bug，保留代码 ✅
      → 如果测试本身有问题，删除测试和代码，用 TDD 重写 ❌

   c) 编译成功，测试通过
      → 这证明测试没有验证任何东西（代码已存在）
      → 删除测试和相关服务，用 TDD 重新实现 ❌
```

### 阶段 2：TDD 重新实现（仅针对失败的测试）

**如果测试发现真实的 bug 或问题：**

```
保留该服务，因为：
1. 测试确实验证了行为
2. 虽然不是 TDD，但测试有实际价值
3. 可以在未来用 TDD 改进
```

**如果测试立即通过或测试本身有问题：**

```
删除该服务，用 TDD 重新实现：

RED:
1. 写一个失败的测试（不参考旧代码）
2. 运行测试，确认失败

GREEN:
3. 写最小代码通过测试
4. 运行测试，确认通过

REFACTOR:
5. 清理代码
6. 运行所有测试，确保全部通过
```

### 阶段 3：集成测试 TDD（2-3小时）

**更新集成测试严格遵循 TDD：**

```
对于每个集成测试文件：
1. WechatCallbackIntegrationTest.cc
2. CreatePaymentIntegrationTest.cc
3. QueryOrderTest.cc
4. RefundQueryTest.cc
5. ReconcileSummaryTest.cc

RED:
1. 写一个失败的测试用例
2. 运行，确认失败（功能缺失）

GREEN:
3. 更新控制器/服务代码以通过测试
4. 运行，确认通过

REFACTOR:
5. 清理
6. 运行所有测试
```

## 决策矩阵

| 测试状态 | 测试价值 | 行动 |
|---------|---------|------|
| 编译失败 | 无法判断 | 修复测试编译，重新评估 |
| 运行失败（发现真实 bug） | 高 ✅ | 保留代码，标记为"测试后添加，需 TDD 重写" |
| 运行通过 | 无 ❌ | 删除代码和测试，用 TDD 重新实现 |
| 测试本身有问题 | 无 ❌ | 删除代码和测试，用 TDD 重新实现 |

## 成功标准

### 最低标准（可接受）
- [ ] 主应用编译通过
- [ ] 所有单元测试编译通过
- [ ] 至少 50% 的单元测试能发现真实问题
- [ ] 集成测试用 TDD 更新

### 理想标准（TDD 合规）
- [ ] 所有服务都用 TDD 重新实现
- [ ] 每个测试都在实现前失败过
- [ ] 集成测试用 TDD 更新
- [ ] 100% 测试通过率

## 时间估计

| 任务 | 最低标准 | 理想标准 |
|------|---------|----------|
| 测试发现 | 1 小时 | 1 小时 |
| 代码验证 | 1-2 小时 | 6-10 小时 |
| 集成测试 TDD | 2-3 小时 | 2-3 小时 |
| **总计** | **4-6 小时** | **9-14 小时** |

## 风险和缓解

### 风险 1：所有测试都立即通过
**影响：** 证明现有测试无价值
**缓解：** 删除所有服务代码，用 TDD 重新实现（9-14小时）

### 风险 2：测试发现设计问题
**影响：** 需要重新设计服务接口
**缓解：** TDD 重新实现，测试会引导更好的设计

### 风险 3：时间不足
**影响：** 无法完成完整 TDD 重新实现
**缓解：** 优先验证关键服务（PaymentService, CallbackService）

## 建议

**立即可执行：**

1. 花费 1 小时运行现有测试
2. 根据结果决定：
   - 如果测试有效 → 保留代码，标记为技术债务
   - 如果测试无效 → 删除代码，用 TDD 重新实现
3. 集成测试更新严格遵循 TDD

**长期计划：**

- 将非 TDD 代码标记为技术债务
- 在未来的维护中用 TDD 逐步重写
- 新功能开发严格遵循 TDD
