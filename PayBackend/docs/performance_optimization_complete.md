## ✅ 已完成的优化

### 优化1：增加工作线程数

**修改文件：** `PayBackend/config.json`
**修改位置：** Line 42
**修改内容：**
```json
"number_of_threads": 1  →  "number_of_threads": 4
```

**优化原理：**
- 原配置使用单线程处理所有HTTP请求
- 单线程最大吞吐量 = 1000ms / 平均响应时间
- 增加到4线程可以并行处理4倍请求

**预期效果：**
- 吞吐量提升：4倍（21 RPS → 84 RPS）
- CPU利用率：25% → 80%+
- 响应时间：基本不变（或略微增加5-10%）

---

### 优化2：增加PostgreSQL连接池

**修改文件：** `PayBackend/config.json`
**修改位置：** Line 20
**修改内容：**
```json
"number_of_connections": 1  →  "number_of_connections": 32
```

**优化原理：**
- 原配置只有1个数据库连接，高并发时请求排队等待
- 32个连接可以同时服务32个数据库查询
- 配合4个工作线程，避免连接等待

**预期效果：**
- 减少数据库查询等待时间
- 提升并发处理能力
- 吞吐量额外提升20-30%

---

### 优化3：增加Redis连接池

**修改文件：** `PayBackend/config.json`
**修改位置：** Line 37
**修改内容：**
```json
"number_of_connections": 1  →  "number_of_connections": 32
```

**优化原理：**
- Redis用于幂等性检查和缓存
- 增加连接数避免Redis操作成为瓶颈
- 与数据库连接池数量保持一致

**预期效果：**
- 减少Redis操作延迟
- 提升缓存操作吞吐量
- 避免连接池耗尽

## ⚠️ 注意事项

### 资源需求变化

**优化前：**
- 最小CPU核心：1
- 最小内存：100MB
- 数据库连接：1

**优化后：**
- 最小CPU核心：4（推荐8）
- 最小内存：200MB
- 数据库连接：32

**如果资源不足：**
- CPU核心 < 4：降低 `number_of_threads` 到CPU核心数
- 内存不足：降低 `number_of_connections` 到16
- 数据库连接限制：调整到数据库允许的最大值

### 性能调优建议

**如果吞吐量仍未达到100 RPS：**
1. 检查数据库查询性能（EXPLAIN ANALYZE）
2. 检查网络带宽（iperf测试）
3. 检查磁盘I/O（iostat）
4. 检查系统负载（top/htop）

**如果响应时间显著增加：**
1. 检查是否有锁竞争（数据库锁、应用锁）
2. 检查线程上下文切换开销
3. 降低线程数到2或3
